================================================================================
1. SECURITY
================================================================================

1.1 — .env.local CONTAINS A VERCEL OIDC TOKEN — CRITICAL
  website/ahmed-site/.env.local contains a VERCEL_OIDC_TOKEN JWT. This
  is a time-limited Vercel deployment token. While .env.local IS in
  .gitignore, the file currently exists on disk with a real token.
  Confirm this has NOT been committed to git history:
    git log --all --full-history -- .env.local
  If it was ever committed, rotate the token immediately via Vercel
  dashboard. Also check the `vercel/install-vercel-speed-insights-4fg4gw`
  branch in .git/refs.

  ACTION: Delete this token or confirm it's expired. Never store OIDC
  tokens in .env.local — they should only exist in CI/CD environment.

1.2 — IN-MEMORY RATE LIMITER DOES NOT SURVIVE COLD STARTS
  src/lib/ratelimit.ts uses a `Map<string, Entry>` in the Node.js process
  memory. On Vercel serverless functions, cold starts create a new
  process with an empty map, completely resetting the rate limit. An
  attacker can send bursts of 5 requests, wait for a cold start, and
  repeat indefinitely.

  ACTION: Use an external store (Upstash Redis, Vercel KV) or at minimum
  implement a sliding-window algorithm. Document the limitation clearly.

1.3 — CSP HAS 'unsafe-inline' FOR SCRIPTS AND STYLES
  next.config.ts CSP allows `script-src 'self' 'unsafe-inline'` and
  `style-src 'self' 'unsafe-inline'`. This is required by Next.js for
  development and in some production scenarios. However, the CSP comment
  says "unsafe-eval removed (not needed in production)" — verify this
  is actually true by testing a production build with strict CSP. Consider
  using `'strict-dynamic'` with a nonce or hash approach if feasible.

1.4 — CONTACT FORM LACKS CSRF PROTECTION
  The /api/contact route has rate limiting and Zod validation but no
  CSRF token or SameSite cookie check. An attacker can craft a form on
  another domain that POSTs to the endpoint. While the honeypot provides
  partial protection and data is only emailed, add:
  - A SameSite=Strict session cookie with a CSRF token
  - OR verify the Origin/Referer header matches the production domain

1.5 — HARDCODED DEFAULT EMAIL ADDRESS
  src/lib/resend.ts defaults TO to 'ahmedyhussain07@gmail.com' if
  CONTACT_TO_EMAIL is not set. This makes the email address visible in
  source code. While it's already publicly visible on the site, consider
  throwing an error instead of falling back to a hardcoded address to
  ensure production configuration is deliberate.

1.6 — RATE LIMITER setInterval UNREF IS NODE-SPECIFIC
  ratelimit.ts: `.unref?.()` uses optional chaining for the unref method,
  which is Node.js-specific. In edge runtime (which Next.js can use),
  this may behave unexpectedly. Test the rate limiter on edge runtime
  if migrating.

1.7 — ESLINT IGNORED DURING BUILD
  next.config.ts: `eslint: { ignoreDuringBuilds: true }` — the comment
  says "ESLint v9 flat-config / legacy .eslintrc.json compatibility
  issues." This is a workaround. The proper fix is to update the ESLint
  config to flat config format (eslint.config.mjs) or downgrade ESLint.
  Skipping lint during production build means errors that should be caught
  at build time only surface at runtime or in separate CI steps.

2. UI / UX
================================================================================

2.1 — STALE COPY: "TWO SMALL THINGS" BUT THREE GAMES EXIST
  CONTENT.md and the English dictionary (en.ts, games section) reference
  "two" games and "two small things" in several places. Three games now
  exist: Typing Test, Breakout, and The Clause Game. Update ALL references:
  - en.ts → games.intro
  - en.ts → games.meta_description
  - CONTENT.md → Games hub intro
  - page metadata description

2.2 — LANGUAGE TOGGLE: NO FLAG OR VISUAL INDICATOR
  The LanguageToggle component swaps between EN and FR. On mobile, it's
  just a button with text. Consider showing a small globe icon or the
  current language code for clarity.

2.3 — MOBILE MENU: NO SWIPE-TO-CLOSE GESTURE
  The mobile drawer opens with a nice animation but only closes via
  button click or route change. Swiping left/right or tapping the
  overlay backdrop (which doesn't exist) should close the menu.
  Add a backdrop overlay with `onClick={() => setMenuOpen(false)}`.

2.4 — CONTACT FORM SUCCESS "SEND ANOTHER" BUTTON IS SUBTLE
  The "Send another message" button after form success is styled as a
  small text underline link. It should be more visually prominent
  (a secondary button) since it's the only action on the success state.

2.5 — GAMES PAGE: "PLAY" BUTTONS HAVE NO HOVER PREVIEW
  Game cards on the hub page have a "Play" link but no preview image,
  screenshot, or animated thumbnail. Consider adding a small screenshot
  or animated GIF preview for each game (similar to the beam project's
  preview GIFs).

2.6 — CODE PAGE: GITHUB RATE LIMIT FALLBACK IS ABRUPT
  When the GitHub API is rate-limited, the entire page shows a fallback
  message. A better UX: show cached repository data with a "last updated"
  timestamp, then show a subtle banner about rate limiting rather than
  replacing all content.

2.7 — NO SEARCH / FILTER FOR PROJECTS
  With multiple projects and games, a search bar or tag filter on the
  projects/games hub pages would improve discoverability.

2.8 — SCROLL ANIMATIONS: ALL SECTIONS ANIMATE AT ONCE
  SectionReveal triggers a staggered animation but all sections animate
  on initial load regardless of scroll position. Use IntersectionObserver
  to animate sections as they scroll into view (Framer Motion's
  `whileInView`).

2.9 — THREE.JS CIRCUITMESH HAS NO LOADING STATE
  CircuitMesh renders a WebGL canvas as a site-wide backdrop. If WebGL
  is unavailable or slow, there is no fallback — the background is empty
  with no indication. Add a fallback (CSS gradient or static SVG) and
  an error boundary that catches WebGL failures.

2.10 — 404 PAGE IS MINIMAL
  src/app/not-found.tsx shows "Page not found" with a single link home.
  Add links to popular sections (Projects, Games, Tutoring) and a search
  suggestion. The "Return home" button could be more inviting.

2.11 — COOKIE-BASED LOCALE MAKES ALL PAGES DYNAMIC
  Because `getLocale()` reads cookies, every page calling
  `getDictionary()` opts into dynamic rendering. This means NO static
  generation at build time — every page is server-rendered per request.
  Consider Accept-Language header detection for initial load with the
  cookie as an override, and use middleware for the locale cookie check
  to keep pages static where possible.

3. DESIGN
================================================================================

3.1 — "HERO-GRID" CSS CLASS IS UNUSED
  globals.css defines `.hero-grid` but it is not used in any component.
  Either remove it or apply it to the hero section background.

3.2 — CIRCUIT BACKDROP SVG FILES HAVE TYPOS IN FILENAMES
  website/svgs/ directory contains files with typos:
  - "circuit-backgrounf-bottom-right-croner.svg"
    → should be "circuit-background-bottom-right-corner.svg"
  - "ciricle-with-t-right.svg"
    → should be "circle-with-t-right.svg"
  These are used by CyberSigils.tsx; fix the filenames and update imports.

3.3 — GOLD/AMBER ACCENT COLOR ABSENT BY DESIGN
  The monochrome design is elegant, but there are no accent colors for
  interactive states (links, buttons, active nav). Even a subtle
  color for visited links or focus rings would improve scannability
  without breaking the monochrome constraint.

3.4 — FOOTER AND HEADER SHARE NO DESIGN TOKENS
  The Header and Footer have independent styling (border colors, padding).
  Extract shared layout tokens (container width, nav height, border color)
  into the Tailwind config for consistency.

3.5 — THREE.JS PARTICLES COLOR IS HARDCODED
  CircuitMesh particle color (`0.22, 0.27, 0.33`) is inlined. Expose as a
  constant or read from CSS custom properties so the mesh respects theme
  changes (if themes are ever added).

4. CODE HYGIENE
================================================================================

4.1 — NO AUTOMATED TESTS
  There are no test scripts in package.json — no vitest, jest, or
  playwright. The project has zero test coverage. Add at minimum:
  - Unit tests for game engines (breakout-engine, contract-engine, wpm)
  - Unit tests for validation schemas (validations.ts)
  - Unit tests for conversion libraries (bases.ts, bitwise.ts)
  - A basic smoke test for the contact API route

4.2 — NO PRE-COMMIT HOOKS / CI CONFIG
  Similar to armoire, there's no husky/lint-staged setup and no
  GitHub Actions CI. The README lists pre-deploy checks as manual steps.

4.3 — ESLINT CONFIG FORMAT MISMATCH
  .eslintrc.json uses legacy format but package.json has eslint ^9.0.0
  which uses flat config by default. This is why `ignoreDuringBuilds`
  was needed. Migrate to eslint.config.mjs (flat config).

4.4 — DEAD CODE: SVGS DIRECTORY
  website/svgs/ contains 8 SVG files but CyberSigils.tsx seems to use
  only 2 (tl.svg, br.svg?). The other SVGs may be unused. Audit and
  remove dead assets.

4.5 — CLIENT COMPONENTS FETCH THE ENTIRE DICTIONARY
  `useT()` returns the full Dictionary object. Client components only
  use a subset of keys (e.g., Header only needs `t.nav.*`). This means
  all ~10KB of i18n text is serialized into every client component
  bundle. Consider splitting the dictionary or using a context selector
  pattern (like Zustand selectors but for i18n).

4.6 — CODE DUPLICATION WITH ARMOIRE PROJECT
  Both armoire and website share identical:
  - Tailwind color tokens (background, foreground, surface, etc.)
  - Font configuration (Inter + Playfair Display)
  - Motion variants (motion.ts files are very similar)
  - Tailwind keyframes (fade-in, slide-up)
  - CSS reset/base styles
  Consider extracting shared design tokens and utilities into a
  private npm package or a monorepo setup.

4.7 — INLINE EMAIL ADDRESS IN MULTIPLE LOCATIONS
  'ahmedyhussain07@gmail.com' appears in:
  - src/lib/resend.ts (default TO)
  - .env.example (default)
  - Footer.tsx (hardcoded)
  - en.ts dictionary (footer, contact, legal pages)
  - fr.ts dictionary
  - sitemap.ts
  Centralize in a single constant or environment variable.

4.8 — SENDCONTACTEMAIL DOES NOT HTML-ESCAPE MESSAGE
  src/lib/resend.ts sends the message as plain text, which is fine.
  However, the `from` field uses template literal:
    `[ahmedyhussain.com] ${subject}`
  If a user includes `\n` or special chars in the subject, it becomes
  part of the email header. Add basic sanitization of subject before
  inserting into the email subject line.

4.9 — LOCAL STORAGE GAME DATA: NO NAMESPACE COLLISION PREVENTION
  src/lib/games/storage.ts uses `ahmed-site:games` prefix which is good,
  but there's no versioning. If the storage format changes, old data
  could cause issues. Add a schema version number.

4.10 — i18n: fr.ts EXISTS BUT MUST BE VERIFIED COMPLETE
  The French dictionary (fr.ts) should be checked for completeness.
  Every key added to en.ts must have a French translation. TypeScript
  enforces this through type inference, but the translations themselves
  need human review for accuracy.

5. WORDING / GRAMMAR
================================================================================

5.1 — INCONSISTENT VOICE: "I" VS "AHMED HUSSAIN"
  Privacy Policy, Tutoring, About use first person ("I am"). Terms of
  Use and some footer copy use third person ("Ahmed Hussain"). Choose
  one voice and use consistently:
  - Professional pages (Terms, legal): third person
  - Personal pages (About, Tutoring): first person
  OR unify to all first person (more personal portfolio approach)

5.2 — "CURRENCY" AMBIGUITY
  Terms of Use: "I make no representations as to the completeness or
  currency of information provided." In Australian English, "currency"
  means "up-to-dateness" (correct usage). However, international readers
  may read it as monetary currency. Consider: "...completeness or
  timeliness of information provided."

5.3 — "TWO" VS "THREE" GAMES — CONSISTENCY UPDATE NEEDED
  As noted in UI/UX, all references to "two" games must be updated to
  "three" or rephrased to "a few small games" for future-proofing.

5.4 — HOMEPAGE META DESCRIPTION PUNCTUATION
  "working where law meets computing and the governance of artificial
  intelligence" —  the lowercase after comma is grammatically correct
  (continuation of sentence), but the fragment reads awkwardly. Consider:
  "Personal website of Ahmed Hussain, BCom/LLB(Hons) candidate at ANU.
  Law, computing, and the governance of artificial intelligence."

5.5 — CONTACT SECTION "I WOULD BE GLAD TO HEAR FROM YOU"
  This is grammatically correct but slightly stilted. More natural:
  "I'd love to hear from you" or "I'd be glad to hear from you."

5.6 — PRIVACY POLICY: "REASONABLE TECHNICAL AND ORGANISATIONAL SECURITY MEASURES"
  This is boilerplate legal language and is fine, but consider adding a
  brief layperson explanation: "This means I use industry-standard
  practices to keep your data safe."

5.7 — FOOTER BLURB IS DEFENSIVE
  "All content on this website is copyrighted and may not be reproduced,
  scraped, indexed for AI training, or incorporated into generative AI
  systems without prior written permission." — This is prominent but
  reads as defensive/combative for a personal portfolio. Consider moving
  the detailed AI prohibition to the Terms page and using a shorter,
  friendlier footer copyright line.

6. PERFORMANCE
================================================================================

6.1 — CIRCUITMESH RUNS CONSTANTLY
  The Three.js CircuitMesh renders a continuous animation loop even when
  the tab is not visible. Use `requestAnimationFrame` with
  `document.hidden` check or reduce the render rate when the tab is
  in the background.

6.2 — NO IMAGE OPTIMIZATION FOR LAWYER.JPG
  public/lawyer.jpg is served as-is. Next.js Image component with the
  `images` config optimizes images, but verify this image is referenced
  via `<Image>` (not `<img>`) to get automatic optimization.

6.3 — FONTS BLOCK RENDERING
  Inter and Playfair Display use `display: swap` which is good, but
  they're loaded from Google Fonts via next/font. The layout shift when
  fonts swap can be noticeable. Set explicit `size-adjust` or use
  `adjustFontFallback: false` and match fallback font metrics.

6.4 — GAME ENGINES MUTATE STATE DIRECTLY
  breakout-engine.ts functions like `stepPhysics(state, dt)` mutate the
  passed state object. This is imperative and performant but makes
  debugging and time-travel harder. Consider an immutable approach with
  returned new state objects (measure performance impact first).

6.5 — REACT THREE FIBER BUNDLE SIZE
  @react-three/drei and @react-three/fiber add significant bundle size
  (~150KB+). They are only used on the Silicon page. Verify these are
  code-split and lazy-loaded (next/dynamic with ssr: false).

6.6 — GITHUB API CALL HAS NO CACHING
  src/lib/github/repos.ts fetches live from GitHub API with no caching.
  Vercel's Data Cache or `next/cache` `unstable_cache` should be used
  to cache the response for at least 5–15 minutes to avoid rate limits
  and reduce TTFB.

7. ACCESSIBILITY
================================================================================

7.1 — THREE.JS CANVAS HAS NO ACCESSIBLE ALTERNATIVE
  CircuitMesh and SiliconAtom are purely visual with no text alternatives
  for screen readers. Add `aria-hidden="true"` to the canvases and
  provide text descriptions.

7.2 — FOCUS TRAP IN MOBILE MENU IS WELL IMPLEMENTED
  Header.tsx mobile menu focus trap is good. Add `aria-label` to the
  mobile nav element explicitly labeling it as "Mobile navigation."

7.3 — COLOR CONTRAST: MUTED TEXT ON BACKGROUND
  `text-muted-foreground` is #a1a1aa on #09090b background. This has a
  contrast ratio of ~7.8:1 which exceeds WCAG AAA. Good.

7.4 — SKIP LINK SHOULD TARGET CORRECT ID
  Layout has `<a href="#main-content">Skip to main content</a>` and
  `<main id="main-content">` — this is correctly aligned. Good.

7.5 — GAMES LACK KEYBOARD-ONLY INTERACTION
  Breakout uses mouse/touch/keyboard, but Contract Game and AGLC4
  Generator use chip selects that should be navigable via Tab and
  Enter/Space. Verify all interactive elements are keyboard-accessible.
